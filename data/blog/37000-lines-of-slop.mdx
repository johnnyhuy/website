37,000 lines of code a day isn't a flex. It's a warning.

Garry Tan claimed that number across five concurrent projects. Tech press reported it as proof of AI's power. A developer called Gregorian actually audited one project. Found something very different.

## The Audit Results

| Category | What They Found |
|----------|-----------------|
| Test files | 28 test files bundled into production bundle, ~300KB sent to every visitor |
| Scaffolding | A literal Rails Hello World controller shipped in production |
| Images | One PNG at 2MB, another at 1.99MB, zero modern image handling |
| Editor | 520KB rich text editor speculatively shipped backend to frontend |
| Accessibility | 47 images with no alt tags |
| Rendering | Page content rendered twice — once mobile, once desktop |

Site loads. Scores 80 on Lighthouse. But "it works" is a dangerously low bar.

Users on slow connections, screen readers, older devices — they bear the real cost. Digital landfill.

## Line Count Is a Terrible Metric

Raw line counts are meaningless without context. 37,000 lines per day isn't 37,000 lines of validated, production-ready code. It's a river of text containing dead code, duplication, accessibility failures, architectural nonsense. Without human review, nobody knows what's actually in there.

> You sure as hell don't know what is going into your code base if you're not reading it.

Lines of code has always been a poor productivity measure. AI just made it possible to generate meaningless lines at industrial scale.

## Slow Down and Review

Mario Zechner, creator of the Pi agent framework, says the answer isn't abandoning AI. It's using it deliberately.

His argument: let AI handle the boring, repetitive stuff that teaches you nothing. Then carefully evaluate what it produced.

Practical guidance:
- Set daily limits on how much AI-generated code you accept, calibrated to your actual review capacity
- Don't accept code you can't read and understand
- Use AI for mechanical coding, not as an autonomous agent without oversight

## Tooling for Hygiene

Static analysis tools like Fallow surface what humans miss:
- Dead code
- Duplicate lines
- Circular dependencies
- Complexity hotspots

AI agents tend to solve problems locally dozens of times rather than globally once. Tools like Fallow identify where this pattern contaminated the codebase.

## The Core Philosophy

> I'm not outsourcing my brain. I'm using my brain to outsource the work.

Outsourcing your brain = accepting AI output without understanding it. Using your brain to outsource work = directing AI toward mechanical tasks while you handle judgement, review, architecture.

First approach produces slop. Second produces leverage.

## Summary

- **Line count ≠ productivity.** 37,000 unreviewed lines is liability, not achievement.
- **"It works" is a low bar.** Performance, accessibility, maintainability matter.
- **Review every line.** If you can't understand it, don't merge it.
- **Set daily AI limits.** Calibrate to review capacity, not generation capacity.
- **Use tools to catch what you miss.** Static analysis finds patterns humans overlook.

## What's Next

Next time you see someone bragging about AI line counts, ask how many lines were reviewed by a human who understood them. The answer tells you whether they're building software or generating slop.
